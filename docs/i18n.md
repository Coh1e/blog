# Bilingual handling (CN ↔ EN)

The site is fully bilingual. Chinese is the default language, sitting at root paths; English mirrors live under `/en/`. This document is the contract for how that mapping works.

## URL contract

| CN URL | EN mirror |
|---|---|
| `/` | `/en/` |
| `/posts/markdown-blog/` | `/en/posts/markdown-blog/` |
| `/notes/<slug>/` | `/en/notes/<slug>/` |
| `/all/` | `/en/all/` |
| `/tags/<tag>/` | `/en/tags/<tag>/` |
| `/pages/about/` | `/en/pages/about/` |

The slug is identical between languages; only the `/en` prefix differs. This is by design — `LangToggle`'s mirror-URL math depends on it.

If a CN page has no EN translation (or vice versa), `LangToggle` should fall back to the locale homepage rather than 404. (Build-time mirror-existence check is on the Day 4–5 list.)

## Filesystem mapping

```
content/posts/markdown-blog.md       →  /posts/markdown-blog/
content/en/posts/markdown-blog.md    →  /en/posts/markdown-blog/
```

`lib/content.ts` infers `lang: "en"` from any path that starts with `en/` after the `content/` root. Override with explicit `lang:` frontmatter only when the path-based inference is wrong (rare — usually means the file is misplaced).

## Routing

Two parallel route trees in `src/routes.tsx`. Every route has a CN form and an EN form. They render the same page component with a different `locale` prop:

```tsx
{ index: true,    Component: () => <HomePage locale="zh" /> },
{ path: "en",     Component: () => <HomePage locale="en" /> },
```

The page component then maps `locale` to a `lang` (`"zh-CN"` or `"en"`) and a copy bundle. There is no i18n library, no message catalog, no dynamic translation. **English copy is hand-written, in the page component, in plain JSX.** This is intentional — for two languages where every string is human-edited prose, a translation library is overhead.

Long-form content (articles, notes) is *not* translated programmatically — each language has its own markdown file. There is no "auto-fallback to CN if EN missing" today.

## `LangToggle` mechanics

`LangToggle` takes a `currentLocale` (`"zh"` or `"en"`) and an optional `pageSlug` (the path **without** the locale prefix — `""` for the home, `"posts/foo"` for an article). It renders two `<Link>`s:

```ts
const zhHref = pageSlug ? `/${pageSlug}/` : "/"
const enHref = pageSlug ? `/en/${pageSlug}/` : "/en/"
```

The active link gets `aria-current="true"` and a different visual style. Clicking the inactive link navigates to the mirror URL, which then re-renders the page in the other language because `routes.tsx` has separate entries for both.

`pageSlug` is computed by the page component based on its route — for the home it's `""`, for an article page it's the entry's slug minus the locale prefix.

## `<html lang>`

`App.tsx` reads `useLocation().pathname` and sets `document.documentElement.lang` to `"en"` or `"zh-CN"` accordingly. This is what screen readers and search engines use to identify the page's language.

## Mixed-script paragraphs

Body text uses a stack font: `EB Garamond` (Latin) → `LXGW WenKai` (CJK fallback). The browser picks per glyph, so a paragraph like "我刚读完了 *On Writing Well*" renders Latin words in Garamond and CJK in 文楷 with no markup needed.

## SEO

For bilingual sites, search engines need explicit hints about the language pairing:

- Each page should emit `<link rel="alternate" hreflang="zh-CN" href="...">` and `<link rel="alternate" hreflang="en" href="...">` pointing to its mirror.
- An `x-default` alternate tells crawlers which language to show users without a strong preference (we use `zh-CN`).
- Per-page `<title>` and `<meta description>` should be in the page's language.

This is wired via vite-react-ssg's `<Head>` component, planned for Day 2 (when articles exist) and refined on Day 5 (deploy). See `architecture.md` § SEO for the full picture.

## What we're not doing

- **No URL-less locale switching** (e.g. `?lang=en` query param). The locale is part of the URL.
- **No automatic browser-language detection.** Users land on `/` (CN) and click `中文 · EN` if they want English. This is deliberate — automatic redirects fight crawlers and confuse direct links.
- **No partial / machine-translated pages.** If an EN page doesn't exist, it doesn't exist. Don't auto-fall-back to CN content with an EN URL.
